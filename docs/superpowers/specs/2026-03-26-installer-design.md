# MYCONEX Installer — Design Spec
**Date:** 2026-03-26
**Status:** Approved

---

## Overview

A cross-platform installer for MYCONEX that supports Linux and Windows, three node roles (hub, full node, lightweight node), interactive TUI by default with SSH and unattended fallbacks, and a fleet-deployment answer-file system.

**Out of scope:** macOS / Apple Silicon support is not covered by this release.

---

## Goals

- Single command gets any machine into the mesh correctly
- Works locally (TUI), over SSH (plain prompts), and in automation (unattended)
- Hub installs the full Docker service stack; nodes install only what their hardware supports
- Idempotent — safe to re-run on failure or upgrade
- Secrets never stored in fleet config files

---

## File Layout

```
install.sh                        # canonical installer entry point (Linux + WSL)
install.ps1                       # Windows wrapper (~50 lines)
install/
  lib/
    ui.sh                         # TUI abstraction (whiptail → plain fallback)
    detect.sh                     # OS, hardware tier, Docker, display/SSH
    profiles.sh                   # component matrix per role/tier
    steps/
      core.sh                     # Python, pip deps, submodules
      hub_services.sh             # Docker Compose hub stack
      llm.sh                      # Ollama pull, llama.cpp, LM Studio wiring
      gpu.sh                      # NVIDIA Container Toolkit, CUDA
      hermes.sh                   # hermes-agent + flash-moe submodules
      registration_agent.sh       # lightweight node: Ollama + announce service
      discord.sh                  # discord.py, token prompt
      integrations.sh             # Gmail, YouTube, RSS, podcast ingesters
      dashboard.sh                # web dashboard
      config.sh                   # writes config/node.yaml + .env
      systemd.sh                  # installs + enables systemd units
  answers.yaml.example            # documented template for unattended installs
```

## Install Flow

```
1. DETECT
   OS + distro, WSL/native, display vs SSH, existing Python/Docker/Ollama
   Hardware: CPU cores, RAM, GPU (nvidia-smi) → tier T1–T4
   Skip steps for components already installed

2. ROLE SELECTION  (TUI menu or --role flag)
   > Hub
     Full node
     Lightweight node (Ollama only)

3. FEATURE SELECTION  (varies by role)
   Hub:          mandatory checklist shown, optional checkboxes
   Full node:    tier-detected plan shown, confirm or override
   Lightweight:  confirm Ollama model for detected tier + registration agent

4. CONFIGURE
   Node name (default: hostname)
   Hub address for nodes (blank = mDNS auto-discovery)
   Optional API keys (Discord token, OpenRouter key, etc.)

5. PLAN PREVIEW
   Show full install checklist with estimated time
   Proceed? [Y/n]

6. EXECUTE
   Progress per step, errors surface inline
   Each step logs to ~/.myconex/install.log
   Each step is idempotent

7. CONFIGURE FILES
   Writes config/node.yaml and .env from collected answers

8. START SERVICES
   Hub:          docker compose --profile full up -d
   Full node:    systemctl enable --now myconex-node
   Lightweight:  systemctl enable --now myconex-registration

9. VERIFY
   Health checks on all installed services
   Summary: what's running, ports, mesh discovery status
```

Hub installs **always** use `--profile full`. Running without `--profile full` brings up mesh infrastructure only (NATS, Redis, Qdrant, Ollama) with no API gateway or LiteLLM proxy — this is valid for development but not a supported hub configuration.

## UX & Interface Abstraction

`ui.sh` exposes four functions used everywhere in the installer:
- `ui_menu` — single-choice selection
- `ui_checklist` — multi-select checkboxes
- `ui_input` — text / secret input
- `ui_confirm` — yes/no prompt

Backend selected once at startup:

| Condition | Backend |
|---|---|
| DISPLAY set + whiptail available | whiptail TUI |
| SSH session or no DISPLAY | plain prompts (read/echo) |
| `--no-tui` flag | plain prompts |
| `--unattended answers.yaml` | silent (no interaction) |

On Windows native, `install.ps1` uses `$Host.UI.PromptForChoice` and `Read-Host` for the same four logical functions.

### Answer File

Generated by `--save-answers --answers-out <path>` (default: `./myconex-answers.yaml`). Consumed by `--unattended`. Running `--save-answers` is interactive but performs no installation — it is a fleet config generator.

```yaml
role: hub                        # hub | full-node | lightweight-node
node_name: ""                    # blank = hostname
hub_address: ""                  # nodes: IP/hostname of hub (blank = mDNS)

features:
  discord: true
  integrations: false
  dashboard: false

api_keys:
  discord_bot_token: ""          # left blank — inject via env var
  openrouter_api_key: ""         # left blank — inject via env var
  nous_api_key: ""               # left blank — inject via env var

ollama_model: ""                 # blank = auto by tier
```

### Secrets in Unattended Mode

Secrets are never written to the answer file. They must be injected via environment variables when running `--unattended`. If a required secret is missing in unattended mode, the installer **aborts immediately** with a descriptive error — it never blocks on a prompt in unattended mode.

```bash
DISCORD_BOT_TOKEN=xxx \
OPENROUTER_API_KEY=yyy \
NOUS_API_KEY=zzz \
install.sh --unattended answers.yaml
```

**Full list of accepted secret environment variables:**

| Variable | Feature | Required when |
|---|---|---|
| `DISCORD_BOT_TOKEN` | Discord gateway | `features.discord: true` |
| `OPENROUTER_API_KEY` | OpenRouter inference | hermes MoE chain |
| `NOUS_API_KEY` | Nous Research inference | hermes MoE chain |
| `GMAIL_CLIENT_ID` | Gmail ingester | `features.integrations: true` |
| `GMAIL_CLIENT_SECRET` | Gmail ingester | `features.integrations: true` |

### Ollama Model by Tier

The installer reads the model for each tier from `core/classifier/hardware.py:TIER_DEFINITIONS[tier]['ollama_model']` — this is the single source of truth. No model names are hardcoded in the installer itself.

**Note:** `hardware.py` currently defines the old model set (llama3.1, llama3.2, phi3). The `config/mesh_config.yaml` was updated to qwen3 models in 2026-03-24 but `hardware.py` was not synced. Updating `hardware.py:TIER_DEFINITIONS` to match `mesh_config.yaml` is a prerequisite task for this installer work (see `docs/DECISIONS.md`).

Current values in `hardware.py` (runtime authoritative until updated):

| Tier | Current model | Target model |
|---|---|---|
| T1 | llama3.1:70b | llama3.1:70b (no change) |
| T2 | llama3.1:8b | qwen3:8b |
| T3 | llama3.2:3b | qwen3:4b |
| T4 | phi3:mini | qwen3:0.6b |

### Systemd Units

| Role | Unit | Exec |
|---|---|---|
| Hub | `myconex-hub.service` | `docker compose --profile full up` (no `-d` — systemd owns the process lifecycle; foreground is required so systemd can track and restart it) |
| Full node | `myconex-node.service` | `python -m myconex --mode worker` |
| Lightweight | `myconex-registration.service` | thin announce agent |

## Platform-Specific Handlers

### Linux (native or WSL)

- Package manager auto-detected: `apt` / `dnf` / `pacman` / `apk`
- Docker installed via official `get-docker.sh` if missing
- NVIDIA Container Toolkit installed via vendor apt/dnf repo if GPU detected
- Python 3.11+ installed via package manager if missing
- Falls back to `~/.myconex/start.sh` launcher if systemd is unavailable (containers, some WSL configs)
- Installs from **`requirements.txt`** at the project root (canonical for CLI installs)

### Windows Native (`install.ps1`, no WSL)

- Installs Chocolatey if missing
- `choco install python git docker-desktop` as needed
- Starts Docker Desktop, waits for engine ready
- Clones/updates repo, runs `pip install -r requirements.txt` (project root)
- Writes Task Scheduler entry for auto-start (systemd equivalent)
- GPU passthrough handled by Docker Desktop — no NVIDIA toolkit install needed

**Note on requirements files:** The project root `requirements.txt` is canonical for all installer paths. `config/requirements.txt` is used internally by the Docker build (`services/Dockerfile`) and is not the same file — the installer must reference the root file on both platforms.

### Windows + WSL (`install.ps1` detecting WSL)

- Pre-flight check: detects WSL version; if WSL1 found, warns user and offers upgrade path (`wsl --set-default-version 2`) before proceeding
- If no WSL: `wsl --install -d Ubuntu`; waits for completion
- If existing non-Ubuntu distro detected: warns user, proceeds anyway (Linux installer auto-detects the package manager)
- Delegates: `wsl bash install.sh "$@"` — full Linux path runs inside WSL
- Docker Desktop WSL2 backend handles service wiring

## CLI Flags

| Flag | Description |
|---|---|
| `--role hub\|full-node\|lightweight` | Skip role selection |
| `--unattended answers.yaml` | Non-interactive install from answer file |
| `--save-answers` | Run interactive, generate answers file, do not install |
| `--answers-out <path>` | Output path for `--save-answers` (default: `./myconex-answers.yaml`) |
| `--no-tui` | Force plain prompt mode |
| `--skip-verify` | Skip post-install health checks |
| `--reinstall` | Force re-run of already-completed steps |
| `--log <path>` | Override log file location (default: `~/.myconex/install.log`) |

## Upgrade Behavior

Re-running the installer on an already-installed machine:

- Each step checks its own completion state (version file or sentinel) before running
- `--reinstall` forces all steps to re-execute regardless of state
- `hub_services.sh` runs `docker compose pull` to update images, then recreates changed containers
- `core.sh` runs `pip install --upgrade -r requirements.txt`
- `config/node.yaml` is never overwritten on upgrade — new fields are merged, existing values preserved
- `.env` is never overwritten — new keys are appended as comments for manual review

---

## Error Handling

- Every step wrapped in `set -e`; on failure: print step name, tail last 20 lines of log, offer retry or skip
- Partial installs are safe to resume — each step checks its own completion state before running
- Network failures on package downloads retry 3× with backoff before surfacing to user
- In `--unattended` mode: any error aborts immediately with non-zero exit code (no interactive recovery)
